# ChatArch CLI Package Conventions

## When To Use

Use this skill when creating, modifying, reviewing, or releasing a ChatArch Python CLI package, especially packages that use or integrate with:

- `chatenv` / ChatEnv typed profiles, env discovery, `new`, `set`, `save`, or profile activation
- `chatstyle` / ChatStyle interactive prompts, confirmations, selections, command input resolution, or text output style
- ChatArch package templates generated by ChatPyPI, ChatUp, ChatTool, or related tooling
- package setup surfaces that should be ChatEnv-discoverable

Load this skill before writing package code, reviewing package diffs, or finalizing a PR that touches ChatEnv/ChatStyle integration.

For broader ChatArch package-development routing, use the ChatMemory theme index `Skills/chatarch/package-development/README.md` when available. It routes new-package, PR/CI, PyPI publisher, release, and extraction tasks to the appropriate focused skills instead of treating any one giant skill as the only entry point.

## Core Principle

Do not reimplement shared ChatArch substrate behavior inside a leaf package.

A leaf package should register with and call ChatEnv/ChatStyle. It should not fork their storage, parsing, interactive, masking, or profile semantics unless the user explicitly asks to change the substrate itself.

If a desired behavior is missing from ChatEnv or ChatStyle, stop and classify it:

1. Is it a package-local adapter around an existing public API? Implement in the leaf package.
2. Is it a general ChatEnv/ChatStyle behavior? Open/update the substrate package instead.
3. Is it a compatibility shim? Keep it small, documented, and tested, and plan deletion.

## Template expectations

ChatArch package templates should generate a real package skeleton, not leave `hello` / `Hello, ChatArch!` as the default user-facing command or release-readiness proof. A template may use a tiny placeholder during scaffolding development, but a package task is not done until the demo command/test/docs are replaced with the package's real CLI surface or explicitly removed.

Generated ChatArch CLI packages should include:

- top-level `--version`
- help text for the actual package command surface
- tests for the real command skeleton and version output
- ChatEnv/ChatStyle wiring when the package owns config or interactive input
- build/check/publish workflow files that match the intended release path

When the user corrects scaffold behavior, update the actual template source and generated-template tests first when possible; do not only patch workflow docs. Keep release skills focused on release gates and route template-internal rules back here.

## ChatEnv Rules

### Register, do not bypass

For a package that owns env/config fields:

1. Provide `src/<module>/config.py` with a `BaseEnvConfig` subclass.
2. Register it in `pyproject.toml`:

```toml
[project.entry-points."chatenv.configs"]
pypi = "chatpypi.config"
```

3. Verify discovery from a clean editable install:

```bash
chatenv --home <task-local-home> list
chatenv --home <task-local-home> test -t <alias> -I
chatenv --home <task-local-home> new <profile> -t <alias> -I --yes
```

### Use ChatEnv's write path

When updating ChatEnv-backed values from a package, use ChatEnv's store primitives instead of rewriting `.env` files yourself:

```python
store = EnvStore(get_paths(home).envs_dir)
values = store.load_active(ConfigClass)
values[FIELD_ENV_KEY] = new_value
store.save_active(ConfigClass, values)
```

For a named profile:

```python
store = EnvStore(get_paths(home).envs_dir)
values = store.load_profile(ConfigClass, profile_name)
values[FIELD_ENV_KEY] = new_value
store.save_profile(ConfigClass, profile_name, values)
```

Do not implement package-local dotenv renderers, profile path logic, quote escaping, interpolation policy, file mode policy, masking, or profile normalization just to update a single key. Those are ChatEnv responsibilities.

If security review raises a concern about ChatEnv file permissions, interpolation, quoting, path traversal, or secret backfill, first inspect the exact ChatEnv `EnvStore` code path and reproduce the issue with a focused regression. Distinguish `EnvStore.load_*`/explicit `save_*` from `BaseEnvConfig.load_from_sources` environment fallback before accepting the finding. If the concern is real and general, fix ChatEnv itself; do not silently patch only the leaf package and create divergent behavior.

See `references/chatenv-explicit-store-writes.md` for the focused review pattern around explicit ChatEnv store writes.

### Profile selection

If a package offers `-e/--env-profile NAME`, define semantics clearly:

- It should read/write the named ChatEnv profile without globally activating it.
- It should not accidentally fall back to a process/default session token when a named profile was explicitly selected.
- For login/refresh flows, profile values should normally take precedence over process env values when `-e` is supplied, so credentials do not cross accounts.
- Error messages should name the profile and key but never print values.

### Sensitive values

- Mark secrets with `is_sensitive=True` in the ChatEnv schema.
- Never print token/password/TOTP/cookie/session values.
- CLI output should show key names, profile names, paths, counts, and booleans only.
- Avoid adding parallel `*_FILE` storage fields when the intended system-of-record is ChatEnv.

## ChatStyle / Interactive Rules

Use ChatStyle for user-facing interactive behavior instead of hand-rolled prompts:

- interactive mode flags should follow the shared `-i` / `-I` / non-interactive pattern where applicable
- use ChatStyle input resolution helpers for commands that can be provided by CLI args, prompts, or defaults
- use ChatStyle confirmation/select/text helpers instead of raw `input()`
- non-interactive mode must fail cleanly when required values are missing
- destructive or publish-like actions need explicit confirmation unless the command has a documented `--yes`/`--force` style override

Package CLIs should expose:

- top-level `--version`
- clear help text for env-var-name options versus secret values
- JSON output mode for automation where useful
- non-zero exits for reserved/planned commands; do not let placeholders look successful

## Dependency Bounds

For ChatArch-owned packages, internal dependencies need upper bounds.

For pre-1.0 internal packages, use the next minor as the compatibility boundary:

```toml
chatenv>=0.2.0,<0.3.0
chatstyle>=0.1.0,<0.2.0
chatup>=0.2.0,<0.3.0
```

For 1.x+ packages, use the next major:

```toml
chatenv>=1.2.0,<2.0.0
```

Do not add direct dependencies on libraries that are only used through ChatEnv/ChatStyle internals unless the leaf package imports them directly. If the leaf package imports it directly, declare it directly with bounds.

## Implementation Checklist

Before finalizing a ChatArch CLI package PR:

1. ChatEnv provider declared in `pyproject.toml` if the package has env config.
2. ChatEnv schema includes every supported field and marks sensitive fields.
3. No package-local reimplementation of ChatEnv file writing/parsing unless explicitly justified.
4. `-e/--env-profile` behavior, if present, is tested for selected-profile isolation.
5. ChatStyle is used for interactive prompts and non-interactive failure paths.
6. Top-level `--version` works and has tests.
7. Runtime dependencies include bounded ChatArch internals.
8. Full gates pass: unit tests, version smoke, build, package check, docs if present.
9. Security scan confirms no hardcoded secrets or raw token/cookie output.
10. Real smoke uses a task-local home or clearly documented existing profile and prints only non-sensitive summaries.

## Common Pitfalls

- Rewriting `.env` files in the leaf package because a single key needs updating.
- Adding `chmod 0600`, custom dotenv rendering, interpolation rules, or profile sanitization in one package instead of fixing ChatEnv globally.
- Treating independent reviewer suggestions as automatically correct without checking the substrate's existing implementation.
- Accepting a ChatEnv secret-backfill finding without distinguishing `EnvStore.load_*` explicit profile-file reads from `BaseEnvConfig.load_from_sources` process-env fallback. Verify the exact call path and run a focused reproduction before changing architecture.
- Letting `-e PROFILE` accidentally read a process env token for a different account.
- Passing secret values in argv instead of env-var-name options.
- Making reserved high-risk operations exit 0 with only a roadmap notice.
- Updating README but forgetting docs, tests, CLI help, or generated templates.

## Verification Commands

Typical local verification:

```bash
PYTHONPATH=src python -m pytest -q
PYTHONPATH=src python -m <module>.cli --version
python -m build
python -m twine check dist/*
mkdocs build --strict  # if docs exist
```

For ChatEnv discovery:

```bash
uv venv --seed <task-project>/playground/env-check
<venv>/bin/python -m pip install -e .
<venv>/bin/chatenv --home <task-project>/playground/chatenv-home list
<venv>/bin/chatenv --home <task-project>/playground/chatenv-home test -t <alias> -I
```
