# Lark CLI Permission Authorization

Use this skill when a `lark-cli` command fails with missing scopes, permission errors, or the user asks for a permission application/authorization link.

## Golden Rule

Do not send `open.feishu.cn/app/.../auth?...` developer-console URLs as if they were the standard user-confirmable authorization link.

For user scopes, the correct user-facing link is generated by `lark-cli auth login --scope ... --no-wait --json` and should look like:

```text
https://accounts.feishu.cn/oauth/v1/device/verify?...
```

The `verify?` link is the one the user can open and confirm.

## User Scope Flow

Use this when the task needs user identity or user-granted scopes, for example creating/fetching/updating documents as the user.

1. Check current auth state:

```bash
lark-cli auth status
```

2. If the CLI is in bot-only strict mode, switch only after explicit user direction:

```bash
lark-cli config strict-mode off
```

3. Generate a fresh split-flow authorization request:

```bash
lark-cli auth login --scope "<scope1> <scope2>" --no-wait --json
```

4. Extract:

- `verification_url` or `verification_uri_complete`: send this raw URL to the user.
- `device_code`: keep it only for this authorization flow.

5. Generate a QR code for the exact URL:

```bash
lark-cli auth qrcode "<verification_url>" --output "reports/<task>-verify-qr.png"
```

6. Send the user:

- raw clickable verify URL, not wrapped in backticks/code formatting
- the QR image
- instruction: after authorization, come back and say it is done

7. After the user confirms, complete the flow yourself:

```bash
lark-cli auth login --device-code <device_code>
```

Do not run `--device-code` in the same turn before the user has seen the link. Do not reuse stale `verification_url` or `device_code`; generate a fresh one each time.

## Bot/App Scope Flow

Use this when running as `--as bot` and the app itself lacks scopes.

Signs:

- Error code such as `99991672`.
- Error says the app has not applied for required scopes.
- Missing scopes are app/tenant scopes, not user OAuth scopes.

Rules:

- Do not run `auth login` for bot identity.
- App developer/admin must enable scopes in the Feishu/Lark developer console or via the official OpenAPI grant link returned by the platform.
- If the CLI/platform returns a permission URL, preserve it exactly. Do not URL-decode, URL-encode, reassemble, or simplify it.
- If the user expects a `verify?` link, they are asking for the User Scope Flow, not a bot/app console URL.

## Link Formatting For This User

Send URLs as raw links, not as inline code or fenced code blocks. The user's client cannot click code-highlighted links reliably.

## Example: Doc Creation + Verification

To create a doc as user and then fetch it back for verification, the scopes may include:

```text
docx:document docx:document:create docx:document:readonly
```

Generate authorization:

```bash
lark-cli auth login --scope "docx:document docx:document:create docx:document:readonly" --no-wait --json
```

Expected URL shape:

```text
https://accounts.feishu.cn/oauth/v1/device/verify?...
```

After the user authorizes:

```bash
lark-cli auth login --device-code <device_code>
```

Then create and verify:

```bash
lark-cli docs +create --as user --doc-format xml --content @reports/doc.xml
lark-cli docs +fetch --as user --doc "<doc-url>" --format pretty
```

## Safety Notes

- Switching strict mode changes the available identity modes; do it only after user direction.
- User identity means CLI may act as the user. Warn the user when switching from bot-only to user-capable mode.
- Store device codes only for the current task; use mode `0600` if writing them to files.
- Do not store authorization links or device codes in durable memory.
